Tutorial 07

Maker vs taker.

Two deterministic-loop templates for the two sides of a Sera market. Deploy liquidity by posting resting quotes (maker), or take liquidity by crossing the spread the moment the rate beats mid (taker). Both ship dry-run, both honor policy caps.

15 min Node loop no LLM in hot path dry-run default
What you'll have at the end

Two bots you can run against the same dedicated wallet: a maker posting two-sided quotes on one pair, and a taker converting on another the instant the edge clears your threshold — each refusing to touch mainnet until you explicitly ack.

Which one do I want?

market-makertaker

They aren't exclusive — run both against one wallet to make on a deep pair and take on a thin one.

Steps

1Deploy liquidity (maker)

The maker runs a cancel-before-place loop: kill stale quotes, read a multi-source mid, and — only if the mid moved enough — post a fresh bid below and ask above.

deploy the maker
cp -r templates/market-maker ~/my-maker && cd ~/my-maker
cp .env.example .env
# set SIGNER_PRIVATE_KEY (DEDICATED wallet), SERA_API_KEY/SECRET
# keep MM_DRY_RUN=true for the first runs
npm install && npm start
the loop (lib/loop.ts)
cancel_all_orders        # clear stale quotes
mid = multi_source_mid   # median of 3 FX sources
if drift < MM_DRIFT_BPS: hold
place_order bid @ mid × (1 − MM_SPREAD_BPS/1e4)
place_order ask @ mid × (1 + MM_SPREAD_BPS/1e4)
Safe by default

MM_DRY_RUN=true logs the bid/ask it would post and submits nothing. It also runs cancel_all_orders at startup (so a crashed run doesn't leave stale quotes) and on Ctrl-C. Mainnet needs MM_MAINNET_ACK=true.

2Take liquidity (taker)

The taker is the mirror image: it polls for an executable rate and fires a conversion the moment the rate beats the reference mid by enough basis points.

deploy the taker
cp -r templates/taker ~/my-taker && cd ~/my-taker
cp .env.example .env
# set SERA_OWNER_ADDRESS, SERA_API_KEY/SECRET
# keep TK_DRY_RUN=true; add SIGNER_PRIVATE_KEY only when going live
npm install && npm start
the loop (lib/loop.ts)
deal = find_deals        # scan markets, diff Sera vs external mid; filter to TK_PAIR
edge = directional(deal.sera_rate, deal.external_mid)
if edge < TK_MIN_EDGE_BPS: hold
get_balances             # inventory guard — skip if you can't fund the spend leg
convert_and_send         # take it (the edge gate is your slippage protection)

3Cap the risk before going live

Both templates forward the same sera-mcp policy caps. These are enforced server-side regardless of your loop knobs — set them before you flip the dry-run switch:

  • POLICY_PRESET=starter — the tightest preset.
  • POLICY_MAX_NOTIONAL_USD — the most a single quote/take can risk.
  • POLICY_DAILY_VOLUME_CAP_USD — your rolling 24h kill switch.

Use a wallet you funded only for the bot, hard-capped to what you can afford to lose. Each template's README has a full Production checklist — work through it before live trading.

Common gotchas

  • Dry-run is the starting line, not a formality. Watch the logged orders/takes until the prices and edges look right. The signer is wired but inert until you set MM_DRY_RUN=false / TK_DRY_RUN=false.
  • Tool shapes vary by sera-mcp version. The taker's find_deals/get_quote parsers accept several field shapes and refuse to act on anything they can't read — confirm a raw response with sera.doctor before going live.
  • Rate limits are real. Sera enforces per-wallet trade/cancel limits (5/s, 2/s). Keep MM_POLL_SECONDS / TK_POLL_SECONDS sane; the maker's 5-min per-order cancel cooldown will 429 if you hammer it.
  • Spread must beat round-trip cost. Run sera.round_trip_cost for your pair and confirm MM_SPREAD_BPS clears it, or you quote at a loss.
